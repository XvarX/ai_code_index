# 游戏服务器项目 Agent 工作规范

## 最高优先级规则

> **禁止在未输出 Plan 且未获得用户明确确认的情况下编写任何代码。**
> 这是一条不可违反的硬性约束。无论任务大小、无论多么简单的改动，都必须先给出 Plan 并等待用户回复确认后才能动手。
> 违反此规则 = 严重错误。

## 项目概况

- **类型**：多进程、单线程游戏服务器
- **开发环境**：Windows（Agent 当前运行环境，使用系统对应命令）
- **部署环境**：Linux（项目代码仅面向 Linux）
- **编码注意**：读取文件使用 UTF-8 编码

---

## 强制工作流

所有代码修改任务必须严格按以下顺序执行，不得跳过任何步骤：

```
需求分析 → RAG查询获取上下文 → Read完整源码 → 检查aimd细节规则（若有）
→ 判断信息是否充分 → 信息不足则询问用户 → 输出详细Plan → 等待用户确认 → 执行实现 → 自检
```

### 各步骤规则

**1. 需求分析**
- 理解要实现的功能、涉及的模块、是否需要修改现有代码

**2. 查询优先级（从精确到模糊）**
- 知道类名/函数名 → `search_symbol(name)` → file:line → Read
  - 搜方法时优先用 `ClassName.method_name`（如 `MonsterManager.spawn_monster`），只有不知道所属类时才用纯方法名模糊搜索
- 知道模块路径 → `module_overview(path)` → 选择符号 → Read
- 以上都不知道 → `search_by_type(query)` 获取线索 → 回到上面
- 禁止凭猜测直接使用模块名或类名，必须通过工具获取准确信息
- 工具返回结果中的 file 字段即为实际路径，直接用于后续 Read
- 若返回的是某个符号的**使用处**而非定义处，优先用 `search_symbol(符号名)` 直接定位定义；仅在同名符号过多需要消歧时才用 `goto_definition(file, line)`
- 中途发现信息不足时，回到查询重新开始，不要直接跳用 Grep

**3. Read 完整源码**
- 使用 Read 工具读取 RAG 返回的完整文件，不得依赖 code_preview 片段写代码
- 注意文件顶部的 import 语句，了解项目实际使用的库
- Read 过程中若遇到不熟悉的函数调用或类引用，**先判断是否与当前待实现功能相关**：
  - **明确相关**（当前任务会直接调用/修改该符号）：用 `search_symbol(符号名)` 定位并 Read
  - **相关性不确定**（属于正在阅读的代码的依赖，但不确定新功能是否需要它）：停下来询问用户是否需要深入了解
  - **明确无关**（只是被读到的代码使用了，但新功能不涉及）：跳过，不追读
- **读代码时要理解每个模块的职责边界**：Read 的目的是搞清楚"这段代码负责什么"，而不是"这段代码怎么写的"。不要因为某个模块恰好做了某件事，就认为新功能也该由它来做

**3.5 检查 aimd 细节规则**
- 项目根目录下若存在 `aimd/` 文件夹，必须先浏览其中的 `.md` 文件列表
- 根据当前需求涉及的模块和功能，筛选出相关性高的 md 文件并 Read 其内容
- aimd 中的规则是对本文件的补充，涉及具体模块的编码规范、设计约束、接口约定等细节，必须遵循
- 若当前需求的改动范围与所有 aimd 文件均无交集，可跳过此步骤，但必须在 Plan 中说明已检查且无相关规则

**4. 信息充分性判断与 SCIP 深挖**

信息不足时，优先使用 SCIP 工具精确定位（SCIP 是确定性查询，可直接执行），以下三种典型缺口对应三个 SCIP 工具：

| 缺口类型 | 对应 SCIP 工具 | 说明 |
|---------|---------------|------|
| 需要查找调用关系 | `get_call_chain(file, line, direction)` | `"outgoing"` 查它调用了谁，`"incoming"` 查谁调用了它 |
| 需要确认某符号的所有使用位置 | `find_references(file, line)` | 列出项目中所有引用该符号的文件和行号 |
| 需要了解关联上下文 | `search_symbol` + `get_call_chain` | 先用符号名定位定义，再查上下游调用链 |

SCIP 查询后仍不充分（如需跨模块搜索、需理解业务逻辑全貌），则必须询问用户是否允许进一步搜索，不要擅自执行 Grep 等操作。

### 必须询问用户的情况（强制）

以下情况**禁止自行推断或猜测**，必须停下来向用户提问：

1. **找不到入口**：不知道新功能该在哪个模块初始化、由谁触发、挂载到哪个流程上
2. **接口不确定**：需要调用某个已有函数或类，但无法通过 RAG/SCIP 确认它的正确用法（参数含义、调用时机、前置条件）
3. **多种实现路径**：同一个需求有 2 种以上合理的实现方式，且无法从已有代码推断出应该选哪种
4. **职责归属不明**：新功能可以放在模块 A 也可以放在模块 B，没有明确的归属依据
5. **涉及业务规则**：需要知道具体的数值、条件、流程等业务逻辑（如"活动什么时候开始"、"奖励怎么发"），而代码中没有明确体现
6. **依赖外部系统**：新功能需要调用项目中从未出现过的接口、服务、协议，无法确认调用方式

**提问格式**：明确说明"我不知道什么"和"我目前的理解是什么"，让用户能快速判断并给出方向。例如：
> "我需要实现活动的初始化逻辑。目前在代码中没有找到活动初始化的标准入口点。请问活动的初始化应该在哪个模块触发？是跟场景初始化一起还是独立流程？"

**5. 输出详细 Plan（强制门槛）**

Plan 是动笔写代码的**唯一准入条件**。未输出 Plan 或 Plan 未获用户明确确认，Agent 不得执行任何写代码操作（包括 Edit、Write 工具调用）。

Plan 必须包含以下板块：
- **新增模块**：文件路径、包含的类/函数、预估代码量
- **修改模块**：文件路径、每个具体改动点的位置和内容、预估改动量
- **可能影响**：使用 `find_references` 确认修改目标的所有引用方，使用 `get_call_chain` 追踪上下游调用关系，将受影响的模块和原因列出
- **不涉及范围**：明确不会修改的内容，标注保持兼容的部分
- **执行顺序**：按步骤列出实现顺序

Plan 中提到的每个文件、每个改动点都必须具体明确，不允许模糊描述。

**6. 等待用户确认（强制阻塞）**

- 输出 Plan 后，Agent 必须以明确的提问方式征询用户同意
- 在收到用户肯定答复之前，Agent 不得调用任何代码编辑工具（Edit、Write）
- 用户的沉默、话题切换、或未明确表态均不算确认
- 只有用户明确回复"可以"、"确认"、"执行"、"开始"等肯定用语，才能视为授权

**7. 执行中发现新改动点**
- 实现过程中若发现 Plan 未涵盖的模块需修改、未预料的副作用、或需要新增依赖，必须立即停止并告知用户
- 发现新改动点时，先用 `find_references` 和 `get_call_chain` 快速评估影响范围，再向用户汇报具体影响
- 得到用户明确允许后才能继续
- **执行中遇到任何"不知道怎么写"的时刻，立即停下来问用户**，不要靠猜测补全代码。典型场景：不知道该调哪个函数、不知道参数怎么传、不知道初始化流程怎么走

## 代码生成规则

### 库与依赖
- 只使用在项目已有代码中观察到的库，不得凭经验假设
- 若需引入新依赖，必须在 Plan 中说明并获得用户确认
- import 风格与项目现有代码保持一致

### 反过拟合：禁止照搬已有代码的组织方式
已有代码的模块划分、初始化位置、调用路径是**历史结果**，不一定是正确的设计。
Agent 写代码时必须遵循以下原则：
1. **新功能的归属由职责决定，不由"旧代码怎么做的"决定**。例如旧项目的 `init` 模块恰好做了玩法初始化，不代表新功能的初始化也该塞进 init — 要看 init 的实际职责是什么、新功能该由谁管。
2. **写新模块前必须确认该模块的职责边界**。如果旧代码中已有同名模块（如 init/manager/loader），先通过 RAG + Read 理解它**实际负责什么**，再决定新功能是否属于它的职责范围。
3. **参考的是接口约定和编码风格，不是代码组织结构**。应该复用的是：函数签名风格、命名规范、错误处理方式、日志格式。不应该复用的是：模块拆分方式、类层次结构、调用顺序。
4. **如果不确定新功能该放在哪里，必须在 Plan 中明确询问用户**，不要自行推断。

### 风格一致性
- 遵循项目现有代码的命名规范、错误处理方式
- 符合多进程、单线程的架构特点，避免引入多线程模式

### 文件操作
- 文件读写必须指定 `encoding='utf-8'`

## 工具参考（三层架构）

### 第一层：符号搜索（精确、零 API 调用）

按名称直接查找，无需语义理解，速度最快。

| 工具 | 解决什么问题 | 参数说明 |
|------|-------------|---------|
| `search_symbol(name, kind="")` | 按名称搜索符号定义 | 支持纯方法名（如 spawn_monster）自动模糊匹配；kind 可选: class/method/function |
| `module_overview(module_path)` | 列出模块中所有类和函数 | 适合了解模块结构 |
| `find_inheritance(name, direction)` | 查找类的继承关系 | direction: parent/children |

### 第二层：代码导航（SCIP 精确定位）

基于预构建索引的行级代码导航（相当于离线 LSP）。所有工具的输入起点是 **file + line**（1-based）。

| 工具 | 解决什么问题 | 参数说明 |
|------|-------------|---------|
| `find_references(file, line)` | 符号在项目中哪些位置被使用？ | 返回所有引用的 file:line |
| `get_call_chain(file, line, direction)` | 函数的上下游调用关系 | direction: `"outgoing"` = 它调用了谁, `"incoming"` = 谁调用了它 |

### 第三层：模糊搜索（RAG，仅用于初始发现）

需要调 embedding API，结果为概率性匹配。仅在前两层无法定位时使用。

| 工具 | 解决什么问题 | 参数说明 |
|------|-------------|---------|
| `search_by_type(query, chunk_type="", module="")` | 自然语言搜索，支持按类型和模块过滤 | chunk_type: function/class_summary/module_summary；module: 模块名如 monster |
| `find_module_summary(module_name)` | 查模块标准流程 | |
| `find_class_summary(class_name)` | 查类职责、方法列表 | |

### 备用工具

以下工具在特定场景下有用，但不作为推荐工作流的一部分：

| 工具 | 场景 | 说明 |
|------|------|------|
| `goto_definition(file, line)` | 同名符号消歧 | 当多个类有同名方法时，用位置上下文精确跳转。日常用 `search_symbol` 即可 |
| `list_symbols(file, kind="")` | 超大文件结构速览 | 大多数情况直接 Read 文件即可 |

**注意：** SCIP 索引为预构建，若目标代码近期有大幅重构，结果可能不准确，需结合 Read 验证。

## 代码自检项

每次生成代码后，逐项确认：

1. 所有 import 均有实际使用，且模块在项目中存在
2. 语法正确，变量名拼写一致
3. 函数调用参数与定义匹配
4. 代码风格与项目一致
5. 文件操作指定了 `encoding='utf-8'`
6. 符合多进程、单线程架构（无多线程代码）
7. 生成代码仅面向 Linux 环境
8. 新功能的放置位置由职责决定，不是因为旧代码中类似功能放在那里
