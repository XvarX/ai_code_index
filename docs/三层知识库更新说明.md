# 三层知识库更新说明

## 更新概述

本次更新将代码知识库从单一函数层扩展为 **三层架构**：模块层 → 类层 → 函数层，解决了 Agent 无法理解"流程化编程"的问题。

---

## 核心改进

### 问题
- ❌ Agent 不知道"先创建场景，再创建 NPC"这种调用顺序
- ❌ 只能搜到单个函数，看不到标准流程
- ❌ 无法理解模块间的协作关系

### 解决方案
- ✅ **模块层**：提供标准流程和入口点
- ✅ **类层**：提供类的职责和核心方法
- ✅ **函数层**：提供完整代码实现

## 配置说明

### config.yaml 新增配置

```yaml
module_analysis:
  # 强制作为整体分析的顶级模块
  force_whole_modules:
    - "scene"
    - "network"
    - "config"

  # 跳过分析的子目录
  skip_submodules:
    - "__pycache__"
    - "tests"
    - "utils"

  # 子模块最少文件数阈值
  min_files_for_submodule: 2
```

### 智能模块检测逻辑

1. **强制整体** → `scene` 即使有子文件夹也作为整体分析
2. **子模块检测** → `gameplay/monster` 如果 ≥2 个文件，作为独立模块
3. **跳过列表** → `__pycache__` 等不作为独立模块
4. **默认行为** → 其他情况归入顶级模块

## 新增 MCP 工具

### 1. find_module_summary
**用途**：查找模块的标准流程和入口点

```python
find_module_summary(module_name="gameplay/monster")
```

**返回**：
```json
{
  "type": "module_summary",
  "module_name": "gameplay/monster",
  "description": "怪物管理模块，负责怪物的创建、删除和状态管理",
  "responsibility": "管理游戏中怪物的完整生命周期",
  "standard_flow": [
    "步骤1: 调用 MonsterFactory.create()",
    "步骤2: 工厂调用 MonsterManager.create_monster()",
    "步骤3: 返回怪物实例"
  ],
  "entry_points": ["MonsterFactory.create", "MonsterManager.create_monster"],
  "key_classes": ["MonsterManager", "MonsterFactory"]
}
```

### 2. find_class_summary
**用途**：查找类的职责和核心方法

```python
find_class_summary(class_name="MonsterManager")
```

**返回**：
```json
{
  "type": "class_summary",
  "class_name": "MonsterManager",
  "description": "怪物管理器类，负责怪物的创建、删除和状态管理",
  "responsibility": "管理怪物的完整生命周期，包括创建、删除、状态更新",
  "key_methods": ["create_monster", "delete_monster", "update_state"],
  "method_count": "5"
}
```

### 3. search_by_type
**用途**：按类型精确搜索

```python
# 只搜索模块概述
search_by_type(query="创建活动流程", chunk_type="module_summary")

# 只搜索类概述
search_by_type(query="怪物管理", chunk_type="class_summary")

# 只搜索函数
search_by_type(query="创建怪物", chunk_type="function")
```

## 使用示例

### 场景：Agent 需要实现"创建怪物"功能

#### 旧方式（有问题）
```
Agent: 搜索"创建怪物"
→ 找到 create_monster() 函数
→ 不知道应该先调用工厂还是管理器
→ 可能写错调用顺序
```

#### 新方式（正确）
```
Agent 1: 查模块概述
→ find_module_summary(module_name="gameplay/monster")
→ 得到标准流程：MonsterFactory.create → MonsterManager.create_monster

Agent 2: 查类概述
→ find_class_summary(class_name="MonsterFactory")
→ 了解工厂的职责和方法

Agent 3: 查具体函数
→ find_function(module="monster", action="create", target="monster")
→ 得到完整实现代码

→ 正确写出代码
```

## FAQ

### Q: 为什么不直接用 RAG 搜函数？
A: 函数级搜索无法理解"调用顺序"和"标准流程"，模块层提供的是"如何使用"的上下文。

### Q: LSP 分析调用关系会慢吗？
A: 会，但只在构建时运行。查询时用的是向量化后的结果，非常快。

### Q: 子模块检测准确吗？
A: 结合了配置文件和智能检测，可以手动调整 `force_whole_modules` 和 `skip_submodules`。

### Q: 支持多语言吗？
A: 当前只支持 Python，架构可以扩展到其他语言。
